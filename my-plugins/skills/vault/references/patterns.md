# Vault — Accumulated Layout Patterns

Distilled, reusable HTML layout patterns from past vault saves, organized loosely by content type.
SKILL.md "HTML creation" greps this file to reuse a matching layout; "Reflection" appends new patterns here (1–2 lines each).

> **개인정보 금지:** 여기엔 *레이아웃 규칙*만 적는다. 종목/수치/거주지/금전/내면 등 개인 구체값은 절대 박지 말 것 — 그건 `$VAULT_PATH`(비공개)에 속하며 이 공개 파일엔 generic 예시로만 쓴다.

Grep by content-type keyword, e.g.:
```
grep -iE 'self-check|카탈로그|how-to|입문|snapshot|addendum|fan-out|tier|책|시계열' references/patterns.md
```

- **설정/스냅샷류는 grid + key-value 카드가 적합.** 정량 데이터(boolean 토글, 값, 모드)는 텍스트로 풀어쓰지 말고 `auto-fill minmax(220px, 1fr)` 그리드의 KV 카드로 배치하면 한 눈에 스캔 가능.
- **이벤트/파이프라인 흐름은 `label → arrow → target` 구조.** 순차적 흐름(예: hook chain)은 좌측 라벨 · 중앙 화살표 · 우측 타겟의 단순 3열로 시각화하면 다이어그램 없이도 흐름이 읽힌다.
- **도메인 입문 글은 `TL;DR → 30초 사전지식 → 분류표 → 핵심 요인 → 항목 카드 → 찬반 2열 → 실행 포인트` 순서가 잘 먹는다.** 독자가 비전문가일 때, 그가 이미 아는 도메인으로 분류 체계를 매핑하면(예: 익숙한 비유) 이해 속도가 크게 올라간다.
- **연속 시리즈 정리는 마지막에 "이전 글과의 비교 표"를 붙여라.** vault에 이미 비슷한 카테고리 글이 있으면, 새 글 마지막에 `축 × 두 항목` 형태의 6~8행 표로 연결하면 독자가 새 정보를 기존 멘탈 모델에 끼워 넣기 쉽다.
- **기존 vault 항목 업데이트는 새 파일 대신 inline addendum.** 같은 주제의 후속 분석은 새 날짜 파일을 만들기보다 원본에 `<hr>` + 새 `<header>` 블록으로 이어 붙이면 컨텍스트가 한 곳에 모인다. 날짜는 달라도 **같은 날(same-day) 후속도 addendum이 맞다** — 거의-중복 새 파일을 만들지 말 것. 효과를 극대화하려면 (1) 섹션 numbering을 원본에서 이어받고(예: 01~05 → 06~), (2) 원본의 semantic color class(`.verdict/.vrow`, 의미론적 행 클래스, `.calc .pos/.neg`)를 그대로 재사용하면 addendum이 bolt-on이 아니라 한 문서로 읽힌다. 커밋은 그 한 파일만 스테이징 + 메시지에 "addendum" 명시. **이론 → 적용 변종**: 이론 노트가 적용으로 확장될 때는 ① 원본의 의미론적 3색 시스템(안전/기회/위험 등)을 §N+ 매핑 표 행 클래스·룰 트리거 색·verdict 행까지 *끝까지* 끌고 가고, ② 추상 원칙을 구체 수치(임계값·필터·주기 등)로 박아 실행 가능하게. 공식도 `.calc` 블록 한 줄에 구체 수치로 박으면 직관 전달. "추상이 실행으로 닫히는" 단일 문서가 됨.
- **"넓은 카탈로그 doc" + "특정 소스 필터 doc"의 쌍은 자연스러운 시리즈를 만든다.** 한 주제로 broad reference를 한 번 만들고, 이어서 같은 주제를 특정 소스(뉴스 애그리게이터 등)로 필터링한 후속 doc을 만들면, 두 doc이 footer에서 서로 cross-link되어 "전체 풍경 → 국지적 시그널" 두 단계 탐색이 가능. 후속 doc은 source ID를 leftmost 컬럼으로 노출하면 tabular feel + 빠른 navigation.
- **다중 항목 "데이터 + view" entry는 single-source-of-truth + JS derive로 운영 부담을 흡수한다.** 카탈로그·대시보드처럼 항목 카운트·섹션 분류·필터 등 derive 가능한 필드가 많은 entry는 모든 필드를 정적으로 박지 말고 JS가 한 source(`<ul id="src">` 등)에서 계산해 채우게 한다. 항목 추가가 "여러 지점 갱신 → 1줄 추가"로 축소되고 카운트 불일치 risk가 영구적으로 사라진다. 사적/소규모면 JS-off fallback은 ul 한 개라 거의 무료. (vault 리스팅 자체는 Caddy autoindex가 담당하므로 이 원칙은 인덱스가 아니라 개별 entry 레이아웃에 적용.)
- **메타 분석/도입 결정 노트는 "rules 그리드 + 대조 테이블 + decided 카드 + 액션 체크리스트" 조합이 강력.** 외부 도구·라이브러리·스킬을 우리 setup에 흡수할지 검토할 때 ① 외부 규칙을 카드 그리드로 압축(`auto-fit minmax`), ② 우리 현재 상태와 규칙 단위 ✓/⚠ 대조 테이블, ③ `decided` 카드(좌측 녹색 보더 + `.dec-tag`)로 결정 + 근거 한 문단, ④ 즉시 실행 가능한 명령 포함 액션 체크리스트(번호 매김). **decided/open 두 상태로 시각 구분** — 좌측 보더 색(decided=녹/open=주황) + `.dec-tag` status 라벨로 닫힌 결정과 미해결 질문을 분리, 세션 중간 상태도 vault에 남길 수 있어 멀티세션 워크플로우와 잘 맞음. 결정 카드 위에 KV 그리드 `프로필 스냅샷`을 두면 후속 독자가 컨텍스트를 빠르게 복원. **명시적 SKIP 카드를 1급 시민으로** — 외부 사례 비교 시 비교표 마지막 컬럼을 객관 특징이 아니라 **우리 입장**(채택=녹/위임=주황/기각=빨) verdict로 색코딩하고, 기각·생략을 red SKIP 카드로 동등하게 남긴다(근거 한 줄 포함) — 미래의 자기가 "왜 저 좋아 보이는 걸 안 했지?"를 재론하지 않게. 같은 3색을 비교표 verdict·카드 좌보더·done/open 체크리스트에 동기화, 시작 incident는 상단 `.callout`으로 arc 보존, 끝은 dogfood 검증 사실을 done 체크에 박는다.
- **다중 항목/옵션 카드는 "tier 색 시스템 + allocation/score bar + 5필드 micro-row + Phase 시퀀싱" 통합 구조.** 카테고리(예: 1차/2차/실험 또는 S/A/B)에 일관된 색(녹/노/빨)을 CSS 변수로 박고 ① 카드 좌측 보더 ② pill 라벨 ③ 가로 stacked allocation bar 또는 우측상단 점수 뱃지(`tier-s`의 `score` 클래스가 색까지 동기) ④ 점수표 총합 셀(`td.tier-s/.tier-a/.tier-b`) 모두 같은 색으로 동기화. 각 카드는 같은 5필드 micro-row(`<div class="row"><div class="lbl">…</div><div class="val">…</div></div>` 80px+1fr 그리드)를 강제해 N개가 시각적으로 align — 마지막 row(부정 항목)는 점선 separator + 빨강으로 "어두운 면" 자연 구분. 점수표가 있으면 axes 4개 × 5점 = 20점 만점 구조를 그대로 표 컬럼으로 두고 "합계" 컬럼만 tier-색 셀로. Phase 시퀀싱(시간 차원)은 `96px 라벨 | 본문` 2열 stack + **numbered pill cross-ref**(`<span class="pill">P3 …</span>`)로 항목 참조 — anchor 없이도 "이 phase에 어떤 카드가 투입되나" 즉시 읽힘. Phase 끝에 `.mrr`(녹·mono) 결과 한 줄로 못박음. 끝부분에 "이 조건이 트리거되면 thesis 깨짐" empty-checkbox 리스트(`☐` ::before) + 단일 key callout으로 닫아 운용 가능한 문서로.
- **카탈로그 doc은 분류 축이 가치를 좌우한다 — "사용자 셋업과의 매칭" 축이 가장 강력.** awesome-list 형태로 "공식 vs 큐레이션 vs mega-pack"으로 자르면 단순 카탈로그에 그치지만, "structure-match · dual-env match · 도메인 일치"처럼 사용자의 본인 셋업과의 거리로 자르면 actionable해진다. doc 끝에는 반드시 <strong>"옮겨올 implementable picks" 섹션</strong>을 둬서 흥미→행동 다리를 놓을 것 (각 픽에 `<code>아이템</code> <span class="from">from 원본</span>` 구조로 출처와 함께). 사용자가 "이런 거 원했어"라고 다른 축을 신호하면 즉시 rewrite — 카탈로그는 잘못 분류되면 거의 무가치.
- **개념/용어 해부 explainer는 "N개 변종에 일관된 색을 박고 여러 표현(표 → 질문 카드 → verdict 행)에 동기화"하는 패턴이 강력.** 한 용어가 사실 N개의 다른 정의로 쓰임을 보일 때, 각 변종에 CSS 변수 색을 하나씩 배정하고 ① 분류 표의 행 ② "각자 묻는 질문" 카드 ③ 최종 verdict 행에 같은 색을 반복하면 독자가 색만으로 변종을 추적. 정량 비교는 `.calc` 모노스페이스 블록에 색칠한 숫자로 단계별 노출하면 수식 없이도 직관이 전달. 마지막 callout `한 줄 요약`으로 closure. 비전문가용 개념 글의 재사용 템플릿.
- **테제 검증/de-bias(self-check) 노트는 통합 템플릿이 있다 — "테제 callout → SUPPORTS/CONTRADICTS 색코딩 카드 → perception-bias 메타 callout → 보정 테제 + 할인율 split-bar → 증거품질 그리드" 기본 구조에, 시나리오·체류 시간에 따라 §addenda를 얹는다.**
    - **기본 구조**: 살아남은 발견(녹 보더 `.s`)과 무너진 발견(빨 보더 `.c`)을 `SUPPORTS`/`CONTRADICTS` pill로 라벨링, 각 카드에 1차/2차 출처를 `.src` 모노 라벨로 노출. 핵심 장치 둘 — ① **할인율 split-bar**: 원 테제를 몇 % 깎는지를 `유지%`(녹)/`폐기%`(빨) 두 색 바로 수치화(단순 결론보다 강한 closure). ② **증거품질 그리드**: `firm`(1차)/`soft`(2차 미검증)/`disputed`(인과 논쟁)를 상단 보더 색이 다른 KV 카드로 분리해 "근거 자체도 검증했음"을 보여줌.
    - **§ 후속 흡수**: 검증 후 떠오른 연관 직관이 있으면 새 파일 대신 마지막에 `N · 후속` 섹션을 붙여 ① 그 직관이 fear-confirming 서사일 가능성을 `.callout.meta`로 먼저 해체하고 ② 살아남는 변수(녹 `.s`)와 진짜 위험(빨 `.c`)을 같은 SUPPORTS/CONTRADICTS 카드로 판결. 검증된 본문과 후속 탐색이 한 문서에 모임.
    - **§ 결론 뒤집기 + 검증 출력**: self-check가 테제를 깎았는데(예: "도구 A가 설계상 우월") 그 다음 하드 제약(환경 차단 등)이 원래 선택으로 다시 데려가는 경우, 보정 테제(green thesis 블록)와 "제약이 고른 결과"(green callout)를 연속 배치하면 독자가 "동기적 합리화로 되돌아간 것"이 아니라 "환경이 결정했다"를 읽음. 노트가 빌드로 끝나면 묘사 말고 **실제 검증 출력(stdout/JSON 왕복)을 `<pre>`에 verbatim + 행 끝 `✓` 주석**으로 박을 것 — "계약대로 동작"을 말로 하는 것보다 실제 요청/응답 페어가 압도적으로 설득력 있다.
    - **§ action으로 이어질 때 scope 뱃지**: 결론이 실제 변경으로 이어질 땐 "판결 → scope 뱃지 스냅샷 → done/open 체크리스트"로 루프 닫기. 각 변경 항목에 **전파 범위 뱃지** — `repo`(녹, version-controlled → 다른 머신에 따라옴) vs `machine-local`(주황, 머신별 → 수동 복제 필요). dual-env 셋업에선 "이 변경이 다른 머신에 따라오나?"가 가장 자주 묻는 질문이고, 뱃지 하나로 즉시 답된다. 끝의 체크리스트는 `☑ done`(녹) / `☐ open`(주황)로 갈라 "검증·적용 완료 vs 관찰 필요" 분리.
    - **§ 전망/예측 사후검증**: 미래에 외부에서 옳고 그름이 결정되는 thesis는, **"무엇이 관측되면 thesis 측 이김 / 반증 측 이김" 2열 표**를 끝에 박는다. 각 행은 구체적 1차 데이터 지표로 채우면, 미래에 같은 doc을 열어 자기 채점이 된다 — disclaimer/footer 대신 이 표가 들어가는 것만으로 self-check가 일회성 분석에서 *epistemic 자산*으로 격상. 추가로 **"메타 — 내가 복용한 인지적 편향 목록"** 섹션을 `.callout.meta`로 한 단계 더 들어가 perception-bias를 *나의 행동*에 적용 — 다음 self-check 때 같은 함정 회피. 사후검증 표가 외부 사실을 기다리는 장치라면, 편향 목록은 내부 메타인지를 미래 self에게 남기는 장치 — 둘이 짝이 되어야 self-check가 "이번 한 번 잘 깎았다"에서 "epistemic 근육이 누적된다"로 바뀐다.
- **Fan-out 결과 수렴 doc은 "클러스터 strip → 합집합 tier 카드 → 키워드 빈도 표 → 약점 체크리스트" 4단 구조.** 여러 sub-agent가 횡으로 흩어 수집한 raw output을 vault에 다시 모을 때: ① flex-proportional `.strip`으로 클러스터별 발견 수 분포를 즉시 보여주고 ② **클러스터 경계를 무시하고 S/A/B tier로 재정렬**해 큰 카드로 노출(클러스터는 우측 colored tag로만 남김) ③ 클러스터 키워드를 합산한 frequency 표에 "본인 매칭(강/중/약)" 컬럼을 붙이면 객관적 신호와 자기 평가가 동시에 보임 ④ 약점 보완 체크리스트로 closure. tier 카드 안에 `<span class="tag">키워드</span>` pill 1-2개를 메모 앞에 박으면 fit 근거가 즉시 노출. 다음 단계(Phase N)로 넘어가는 시사점을 footer 직전에 짧게 정리. 사용자 의사결정 노트(`-plan.html`)와 결과 수렴 노트(`-shortlist.html`)는 한 쌍으로 짝지어 두면 grilling 세션의 입력/출력이 vault에 보존됨.
- **절차형 how-to/실전 가이드는 "상단 N단계 flow strip → 단계별 섹션 → safe/caution/danger 3색 동기화 → 끝에 액션 체크리스트" 패턴이 강력.** 비전문가 대상 실전 가이드(생활/실무 등)는 ① 상단에 numbered flow strip(`step → step`)으로 전체 흐름을 먼저 보여주고 위험 단계는 색으로 강조, ② 핵심 결론은 맨 위 `key` callout 한 줄로 박고, ③ "이건 안전 / 주의 / 망가짐"을 녹·주황·빨 3색으로 카드·note·인라인 텍스트에 일관 동기화하면 독자가 색만으로 do/don't를 추적, ④ 맨 끝 `☐` 체크리스트로 "오늘 바로 따라 하기"를 닫는다. 전문 도메인 밖(비-기술 도메인) 첫 진입 시에도 잘 작동하는 범용 템플릿.
- **다중 카테고리 "핸드북"은 병렬 fan-out 리서치 → 단일 self-contained doc 합성이 효율적.** 5개+ 카테고리를 각각 sub-agent에 동시 위임(WebSearch 포함)해 구조화 브리프를 받고 한 HTML로 합성한다. ① 상단 sticky `nav.toc`(앵커 칩)로 섹션 점프 ② "우선순위(impact×빈도)" overview strip을 본문 앞에 둬 "어디부터" 답을 먼저 줌 ③ 모든 섹션이 같은 절차형 패턴(앞 룰) 반복 — 단일 semantic 3색(ok/warn/bad)·카드·표·`☐` 체크리스트를 동일하게 써서 N개 이질 도메인이 한 시스템으로 읽힘 ④ 섹션마다 끝에 카테고리 마스터 체크리스트로 닫음 ⑤ 같은 시리즈의 단일 주제 doc이 있으면 cross-link. 브리프는 verbatim 재수록 말고 high-value만 distill. 페르소나·지역·연도 특화 정보는 sub-agent 프롬프트에 명시해 현재성 확보.
- **grilling 노트가 낯선 도메인 교육까지 겸할 땐 "단일 fit-verdict 팔레트(go/sub/no)"를 이질적 섹션 전체에 동기화하라.** 사용자가 "이름만 알지 비교 대상 모름"인 분야의 구매/선택 결정에선, 한 문서가 ① 결정 트레일(Q→pick→why 카드) + ② 도메인 지형도 카탈로그 표를 동시에 담으면 "내 결정 기록 + 분야 입문서" 이중 역할. 핵심 장치: 같은 3색(녹=추천, 주황=보조, 빨=패스)을 (a) tier 리스트 좌측 보더 (b) 카탈로그 표의 마지막 "나한테" 컬럼(`.fit.go/.sub/.no`) (c) how-to 루트 카드에 반복하면 독자가 색만으로 "맞음/보류/스킵"을 전 섹션에서 추적. 카탈로그 표는 객관 특징 컬럼 + **"X ↔ Y 비교 축" 한 줄 mnemonic(`.axes`)**으로 닫으면 "누가 누구랑 경쟁하나"가 압축됨. 결정의 출발점은 끝에 `참고` 섹션 + `.callout.amber`로 보존해 전체 arc를 남긴다.
- **코퍼스/시스템 감사 doc은 "hero stat 카드 → 벤치마크 열 스코어카드 → 자기-반증 인용 → ROI순 promote된 triage" 구조.** 사용자 본인의 데이터(노트 그래프·설정·로그)를 스크립트로 전수 채점한 결과를 남길 때: ① 헤드라인 4개 수치를 `.stat`(big mono 숫자 + bad/ok/warn 색) hero 그리드로 punchy하게 열고, ② 스코어카드 표는 측정값 옆에 **"건강 기준" 열 + 색-코딩 판정**(`.j.bad/.ok/.warn`)을 둬서 "좋은가 나쁜가"가 즉시 읽히게 한다(맨숫자만 나열 금지). ③ 가장 강력한 장치 — **사용자 자신의 산출물을 인용**해 "선언한 방법 vs 측정된 실천"의 괴리를 `.callout`에 박으면 진단이 반박 불가가 된다(머릿속이 아니라 파일이 증거). ④ triage는 문제를 나열하지 말고 **최고 ROI 버킷을 `.tier.hot`(채운 녹색 bg)으로 맨 위에 promote**(나머지는 warn/bad). 단일 stacked `.bar`로 코퍼스 구성(고립% vs 연결%)을 한 줄 시각화하면 hero를 보강. 프로세스 측: "전수검수"류는 N개 통독 금지 — 스크립트로 전 코퍼스를 구조 채점(싸고 100% 커버) 후 플래그된 것 + 방법론-정의 노트만 정독.
- **사용자가 실제로 "적용할" 다중 추천 카탈로그는 항목마다 `붙여넣을 수 있는 config 스니펫 + 명시적 기대효과` 쌍을 강제하고, 상단 요약표에 scope 열을 둬라.** "도구 N개 추천"은 분류 축·단일 도입 결정과 다르다 — 핵심은 *항목별 실행가능성*. 각 카드를 고정 라벨 리듬(`왜 너에게` → `예시`(실제 config/dotfile 스니펫을 `<pre>`에 verbatim) → `기대효과`(결과 문장, 녹) → `주의`(충돌·trade-off, 주황) → `source`)으로 통일하면 긴 카탈로그가 스캔된다. 추천은 *정확한 스니펫*과 *payoff 문장*이 둘 다 있어야 actionable — 둘 중 하나만 있으면 "흥미롭지만 안 함"으로 끝난다. 맨 위 요약표는 `# · 도구 · 한 줄 기대효과 · scope 뱃지`(stow=녹/machine-local=주황) 4열로 — 독자가 detail 카드 들어가기 전에 "뭐가 있고 어디에 적용되나"를 먼저 본다. 도메인별 group 헤더로 카드를 묶고, 채택 안 한 후보는 `honorable mention`에 이유와 함께 보존. 같은 두 축을 이질적 도메인 각각에 적용하는 종합 문서에선 ① 각 축에 CSS 변수 색을 하나씩 박아 inline `.badge`로 만들고 모든 질문 헤더(`Q1 … <badge>축A</badge>`)에 반복 → 도메인이 바뀌어도 축 색은 불변이라 cross-domain 패턴이 색의 반복으로 드러남. ② 그 위에 **축과 직교하는 신뢰도 pill**(`확인됨`=녹 1차, `주장`=주황 2차매체, `추정`=회색)을 개별 claim 끝에 달면, sub-agent가 모아온 출처 품질이 footnote 추적 없이 인라인으로 읽힘. 끝은 `축 색 그라디언트 axis-rule`로 두 축이 섞이는 교차 시사점 섹션을 닫고, footer legend에 두 색뱃지+신뢰도 정의를 재수록.
- **LLM 서브에이전트 배치로 코퍼스를 일괄 분류/백필한 작업의 vault화는 "택소노미 스펙(MECE 멘탈모델) → decided/trap 2색 카드 → 검증 출력 verbatim" 구조가 강력하다.** ① 분류 스펙은 *축 분리*로 박는다 — ME 1차축(가장 안 겹치는 차원) + 다중 태그(cross-cutting: 여러 카테고리를 가로지르면 카테고리 아니라 태그), catch-all이 ~20% 넘고 내부 이질이면 하위 ME축으로 분할(자체 cascade 필수). cascade 번호를 표로 박아 "결정론적 분류 규칙"을 명시하면 LLM 일관성도 올라간다. ② **과정의 진짜 재사용 가치는 *함정*이고 `.dcard.trap`(빨 보더) 카드로 박는다** — (a) 셸 멀티바이트 silent 오작동: zsh unquoted `$var` word-split 안 함, `ls` alias가 경로 오염, `cut/awk+sort/uniq`가 비-ASCII 필드에서 거짓 집계 → 집계·검증은 전부 Python(`yaml.safe_load` 전수). (b) LLM 배치 출력은 키·줄수·필드형식이 안 보장 — 행번호 에코·이름 대신 cascade 번호·빈 컬럼 생략(2-col)·줄 드롭이 실제로 다 났다 → 출력 불신, 입력과 set-equality + position-remap으로 정렬, 번호→이름 결정론 map, 드롭분만 직접 분류. ③ 빌드/검증으로 닫되 실제 검증 출력을 `<pre>`에 verbatim + `✓`(예: 태그 OR 필터가 합집합인지 대조, category facet 일치). 사용자 피드백으로 모델이 v1→v2로 깎이면(용어 rename·축 분할·다중화) 새 파일 말고 같은 노트를 최종 상태로 덮어쓰고 `v2 · 피드백` decided 카드를 한 장 더 얹는다(stale 아카이브 방지).
- **예측/전망 종합 doc은 "확증 fan-out으로 컨센서스 → 반증 전용 fan-out으로 self-check → 할인된 보정 테제로 닫기"의 2단 파이프라인이 표준이다.** 전망류는 출처 풀 자체가 편향(예: 발행자 측 강세)을 가질 수 있어 확증 리서치만으론 위험. 1차로 N축 병렬 에이전트(각 18+ 출처)로 컨센서스/시나리오를 세운 뒤, **반드시 별도 반증 fan-out을 추가로 돌려라** — 최소 (a) "정반대 thesis를 가장 강하게 변호하라"(1차/측정 데이터 우선, 2차·vendor 할인) (b) "그 thesis 주창자들의 이해상충을 매핑하라"(발행/운용 주체 여부, 결과로 누가 수혜하는 구조인가) 두 에이전트. 그래야 인센티브 비대칭이 정량적으로 드러난다(확증 리서치로는 절대 안 나옴). 문서화: 시나리오는 Bull/Base/Bear `.scn` 카드(green/default/red 좌보더 + 우측 mono `prob` pill, 화살표로 확률 재배분 ↑↓ 표기)로 박고, self-check 박스를 같은 문서에 끼우되 **반증 조건을 "검색 전 선등록"으로 명시**하고 각 조건에 `반증됨`(빨)/`유지`(녹) verdict pill을 달아 "4/4 관측됨"처럼 타격을 수치화. 할인율은 확률 재배분(base 40→25%, bear 25→40%)으로 구체화. 과거 짝 doc이 있으면 footer cross-link로 "과거→미래" 시리즈. 핵심: TL;DR에 1차 결론과 할인된 보정 결론을 *둘 다* 박아 "처음엔 이렇게 봤다 → 반증 후 이렇게 깎았다"의 arc를 첫 화면에서 보여줄 것.
- **작동 데모를 "버전 시리즈(v2→vN)"로 진화시킬 땐 각 파일 상단에 `.retro` 회고 배너 한 줄 + lightpanda 헤드리스 검증 + canvas/SVG 동적부 try/catch 방어, 이 3종이 표준 루프다.** 사용자가 "여러 버전 만들어, 이전 거 회고하며 개선"을 요청하면 — ① **각 데모 최상단에 점선 `.retro` 배너**(`⟲ vN-1 → vN 회고 · [직전 약점] → [이번 개선]`)를 박아 진화 서사를 *데모 안에* 보존: 단일 기능 데모들이 "왜 이렇게 바뀌었나"의 사슬로 읽힌다. ② **각 빌드 후 lightpanda(`http.server` 띄우고 goto+evaluate)로 핵심 로직을 즉시 검증** — 엔진 함수 직접 호출(결정론·점수 매핑), DOM 갱신, 탭/필터/정렬, 소셜 루프. **단 lightpanda는 canvas 2D(`createRadialGradient`)와 동적 SVG `innerHTML` 파싱을 미지원** → 그 부분은 검증 불가(실제 Chrome 몫)임을 인정하고, **canvas 그라데이션·그래프는 `try/catch`로 감싸 "장식 실패가 결과 전체를 죽이지 않게"** 방어. ③ **결정론적 의사 엔진**(`fnv` 해시 → `mulberry32` PRNG → 점수)으로 입력당 항상 같은 결과 = 공유 카드 신뢰성. ④ 시리즈 끝엔 통합 버전(허브)이 이전 버전들을 `<a target="_blank">`로 묶는다. **대중 제품은 "익숙한 기존 UI 차용"이 진입장벽을 0으로** — 자체 톤이 멋져도 낯설면 안 쓴다 → 사용자가 이미 매일 쓰는 앱의 레이아웃·인터랙션·색 관습(지역별 관습 차이 주의)을 그대로 모방하고 우리 고유 가치만 익숙한 자리에 얹으면 학습비용 없이 채택.
- **큐레이션 코퍼스의 "시계열 추이" doc은 밀도 불균형을 먼저 고백하고 절대건수 대신 정규화 비중으로 본다 — 그리고 4종 시각화(진자 + heatmap + 날짜 rail + 의도vs결과)를 조합한다.** 시간에 걸쳐 backfill 밀도가 다른 corpus는 연도별 절대건수 비교가 거짓 신호 → 상단 `.caveat`(빨)에 밀도 차이를 명시하고 heatmap을 **열 정규화(연도 내 비중 %)**로 그려 "그 해 무엇이 화두였나"를 본다. 집계는 앞 택소노미 룰대로 Python `yaml.safe_load` 전수(셸 grep/awk 비-ASCII 거짓집계 금지). 시각화 4종: ① **이항축 2색 진자 타임라인** — 한 축을 왕복하는 흐름은 좌보더 색 + `.ph-tag`로 국면을 칠해 "진자운동"이 한눈에. ② **연도내 비중 heatmap** — `data-vals` attribute → JS가 셀 배경 색강도 계산(self-contained, 60%를 max로 클램프), raw % 숫자도 셀에 박아 색+수치 동시. ③ **날짜 rail 타임라인** — 최근 집중 국면은 `70px 날짜 | rail(::before 세로선+node) | 본문` 3열로, 핵심 이벤트는 `.hot`(빨 큰 node)으로 강조. ④ **"의도 vs 실제 결과" 2열 대조 카드** — 좌(의도, dim) / 우(결과, 빨 보더)로 갈라 "의도와 반대로 작동"을 시각적으로 폭로. 같은 도메인 기존 vault 항목이 있으면 footer cross-link.
- **창작/서사 콘텐츠는 "책" 메타포 레이아웃이 분석 노트보다 강력 — full-bleed 표지 → 세리프 리더 본문 → 콜로폰(제작 노트).** 창작/서사 산출물을 vault에 담을 때는 그리드·테이블 대신 ① 100vh full-bleed 표지(표지 이미지를 base64 data URI로 인라인 + 제목/카피/장식 요소를 z-index 오버레이, 하단 scrim 그라데이션으로 가독성 확보) ② `max-width:680px` 세리프 본문(Noto Serif KR, `line-height:1.9`, 장면 구분 `· · ·`, 시점 전환은 pill 헤더, 인-월드 메시지는 좌측 보더 인용 블록) ③ 하단 콜로폰(작품명·플랫폼·장르·핵심 엔진·톤·다음 화 — `dl` grid)으로 구성한다. 콜로폰이 메타데이터를 본문에서 분리 수용하므로 독자의 몰입을 깨지 않으면서도 "이건 왜/어떻게 만들었나"가 한 곳에 남는다. 로컬 이미지는 반드시 base64 인라인(file:// 참조는 self-contained 위배 + html2canvas류 export에서 canvas tainting으로 깨짐) — 편집은 외부참조본, 배포는 인라인본 2파일로 분리하면 반복 수정이 싸다.
- **밈·통념의 "지적 원천"을 파는 explainer는 그 밈의 시각물 자체를 SVG로 정확히 재현하고, 라벨이 사실 어느 학술 이론에서 왔는지 1:1 매핑하라 — "아 이게 그거였어"의 쾌감이 핵심.** 유행 밈/짤의 뿌리 이론을 설명할 때 추상 개념 나열로 끝내지 말고 ① **밈 원본 그래프를 SVG로 그대로 그린다**(예: 사이클 곡선 + 참여자 밴드 + 단계 라벨), ② 밈에 적힌 단어들이 **사실 어느 학자의 어느 차트 라벨인지 출처를 못박는다**. "내가 웃으며 본 밈이 진짜 이론이었구나"의 깨달음이 보상. ③ 이론의 핵심 분류에 일관 3색(녹/주황/빨=안전/주의/위험)을 배정하고 **카드·표 pill·그래프의 참여자 밴드(`<rect opacity=.07>`)까지 동기화**하면 상태 악화가 그래프 위 색 이동으로 읽힌다. ④ 역사 반복은 `사례·시기·'새로운 논리'·결말 pill` 4열 표, 마지막 행에 현재진행(`.pill.s`)을 넣어 과거 패턴이 지금과 연결됨을 보임. ⑤ 비판/한계를 별 섹션(`.crit` 빨강)으로 인정해야 explainer가 신봉이 아닌 렌즈가 된다. 앞서 만든 데모/기획 시리즈의 "초심·지적 출발점"으로 역배치하면 흩어진 작업이 한 뿌리에서 나온 것으로 묶인다.
- **여러 아이디어 방향을 잡는 "ideation 노트"는 ① 폐기된 선행 self-check의 교훈을 상단 meta callout로 먼저 박고 ② 각 아이디어를 lead(녹)/satellite(주황) 좌보더 카드 + micro-row로 분리 ③ "기존 vault 문서 ↔ 새 프레임 요소" 1:1 매핑 표로 알맹이를 증명 ④ 합본 컨셉은 3-layer stack으로 닫는다.** 같은 도메인에서 이미 죽은 thesis가 있으면 새 아이디어 노트는 그 교훈을 `.callout.meta`로 맨 위에 계승해 "왜 이번엔 다른가"를 먼저 답한다 — 핵심 통찰은 보통 **"데이터가 틀린 게 아니라 프레임이 틀렸다 → 같은 입력을 옳은 집으로 이사"**. 각 아이디어 카드는 tier 색을 재사용하되 점수 대신 lead/satellite 2분류(녹/주황 좌보더 + pill)로, 카드 안은 `왜·오류해소·MVP데이터·트래픽` micro-row + 마지막 `.risk`(빨) 행으로 통일. 아이디어가 "빈 포장지"가 아님을 증명하는 장치가 **요소↔문서 매핑 표**(좌열은 새 프레임 용어 mono, 우열은 실재하는 vault 파일을 strong으로 박아 "알맹이는 이미 있다"를 보임). 공유되는 산출물은 `.mock` 결과 카드(good=녹/bad=빨 인라인)로 미리 보여주고, 둘 이상의 아이디어가 합쳐지면 `120px 라벨 | 본문` layer stack으로 "A가 B의 연료가 된다"를 시각화. 끝은 프로토타입 후보 `☐` 체크리스트. 면책은 선행 노트 그대로 계승.
- **보안/remediation 감사 노트는 "감사 doc" 골격(hero stat → 스코어카드 → ROI triage) 위에 decided/SKIP 카드를 얹어 *결정*을 1급으로 남긴다.** 삭제 목록만 적으면 "뭘 지웠나"로 끝나지만, `유지`(녹)/`forward-only`(주황)/`SKIP`(빨) 3-verdict 카드(좌보더 색 + `.dtag`)로 **안 고친 것 + 근거**("이미 박제라 security theater")를 동등하게 박으면 미래의 자기가 재론하지 않는다. hero stat은 **"0(good)=녹 / 제거한 위협 수=빨 / 의도적 잔존=주황"**으로 색을 줘 "보안은 삭제가 아니라 트레이드오프"임을 첫 화면에 전달. 작업 중 새로 **발견한 부수 리스크**(예: 게시 경로 의존성)는 본문에 섞지 말고 끝에 `.callout.alarm`(빨)으로 격리해 강조 — 같은 3색(녹/주황/빨)을 스코어카드 심각도 pill·decided 카드 좌보더·체크리스트 done/open에 동기화.
- **"시장분석 → 기획 제안" doc은 단일 3색(포화=빨/형성중=주황/빈틈=녹)을 진단에서 제안 배치까지 한 줄로 흐르게 하고, "그들의 자리 ↔ 우리 자리" 2열 차별표로 닫는다.** 카탈로그(전수 분류표)만으론 "그래서 뭘 해야 하나"가 안 나온다. ① 먼저 세그먼트를 좌보더 3색 진단 행(`.drow` 150px 라벨 | 본문, st 라벨에 "완전 포화/형성 중/빈틈" mono)으로 압축 — 색이 곧 "들어가지 마/여기다"의 지도. ② 녹색으로 칠한 빈틈을 `.gapbox`(녹 보더) 한 문장으로 못박고, 그 녹색을 그대로 제안의 `lead` 카드 좌보더로 이어받으면 "시장이 빈 곳 = 네가 설 곳"이 색의 연속으로 읽힌다(대안은 `sat` 주황). ③ 제안 카드는 `포지셔닝·대상·차별·목차(numbered .toc)·리스크(빨 라벨 row)` 고정 micro-row로 통일. ④ 차별화는 추상 형용사 말고 **"경쟁자 · 그들의 자리 · 우리 자리" 3열 표**로 박아야 actionable — 각 경쟁 항목마다 우리 위치를 1:1로 좌표화하면 "왜 이게 안 겹치나"가 반박 불가가 된다. 카탈로그가 입력, 3색 진단이 변환, gap→lead 색 연속이 결론.
- **자산/포트폴리오 대시보드(snapshot)는 "hero stat 3카드 → stacked allocation bar → 보유 테이블 → 리스크 경보 카드 → 구조 점검 카드" 골격에, *라이브 마크투마켓 보유와 stale book-value 보유를 반드시 별도 섹션·라벨로 분리*하는 게 핵심이다.** 두 평가 방식(실시간 시장가 vs 마지막 캡처 장부가)을 한 합계로 섞으면 정확도 착시 → hero stat 총액 note에 "라이브 X + book Y"를 명시하고, 보유 테이블도 ① 라이브(손익%·상태 pill 포함) ② book-value(KV 나열, 손익 생략) 두 블록으로 가른다. allocation은 3개 묶음(예: 안전/위험/헤지)을 stacked bar 3색 + 세부 비중표로. 보유별 상태는 유지(녹)/드리프트(주황)/경보(빨) pill, **직전 점검 doc 대비 "권고를 실행했나"**를 구조 점검 카드에 녹/주황으로 동기화하면 일회성 스냅샷이 추적 가능한 시계열 노드가 된다. 리스크는 단일 렌즈(집중도·과열·비용누적 등)별 좌보더 빨 카드. footer에 선행 thesis·전략 doc cross-link.

- **일회성 구체 결정(특정 주문·구매·선택)을 "재사용 결정틀"로 vault화할 때는 ① 핵심 원칙을 *절대평가 거부 → 축별 한 칸 낮추기*로 박고 ② 변수를 N개 `.axis` 카드로 분해 → ③ `.pre` STEP 흐름으로 결정 절차를 박제 → ④ 마지막에 *원래의 구체 결정을 이 틀로 역(逆)검산*하는 "적용 예시" `.checks`(go/no/q 3색) 섹션 → ⑤ "냉장고에 붙이는 한 장 요약" `.pre`로 닫는다.** 개인 선택 상담이 일반 프레임워크로 승격되는 핵심은 **"이게 X해?" 같은 이진 질문을 거부하고 "같은 만족도에서 부하를 어디서 덜까"의 상대·한계조정 프레임으로 재정의**하는 것 — 첫 callout.key(TL;DR)에서 이 전환을 먼저 선언. 변수는 `.axis` 카드(헤더에 `축 N` mono 라벨)로 하나씩 격리하고, 각 축 안에서 가벼움↔무거움을 `.sev hi/mid/lo` 3색 표로 줄세운다. **가장 큰 레버 1개를 본문에서 명시적으로 "최대 레버"라 부르고**, 가장 과소평가되는 레버도 따로 짚으면 우선순위가 선다. 빈도 프레임(`가끔 vs 루틴`)을 meta callout로 일찍 박아 "매번 완벽" 강박을 차단 — 조정은 빈도 잦을 때용임을 명시해야 지속가능. 적용 예시 섹션은 출발 조합을 `<em>`으로 인용 → `.checks`로 각 축 조정을 go(+)/no(×)/q(?)로 검산 → "착지점" callout.key로 닫으면 "추상 틀이 원래 문제를 실제로 푼다"가 증명된다. 사이드/부속 선택은 같은 틀의 축소판(예: 베이스 종류 3색 표)으로 본 틀과 평행하게.
- **운영/스케줄 플레이북에서 "시간축 정렬"이 핵심 메시지면 비율 기반 absolute 타임라인 다이어그램을 박아라.** 시작이 가변적인 자원(윈도우·세션·배치)을 작업 블록에 맞추는 류의 가이드는, 말로 풀면 안 잡히고 그림 한 장이 결론을 즉시 전달한다. 컨테이너 width를 전체 time span = 100%로 잡고, 각 구간 트랙(자원 A / 자원 B / 작업 블록)을 `left:%`·`width:%`로 absolute 배치 — 트랙끼리 같은 시간 좌표계를 공유하므로 "겹침/가로지름"이 눈으로 읽힌다. 임계 경계(리셋·전환점)는 트랙을 관통하는 세로 라인(`.reset`, 위에 라벨)으로, 흘려보내는 구간과 실사용 구간은 색/해치로 구분. 하단 `.tl-axis`에 같은 % 좌표로 눈금을 깔면 추상 비율이 실제 시각에 앵커된다. 절차형 how-to 골격(TL;DR callout → 다이어그램 → 공식 박스 → safe/caution/danger 3색 한계 카드 → ☐ 운영 체크리스트)과 결합하면 "원리→그림→실행"이 한 흐름. self-contained 제약상 폰트는 system 스택, 색은 light/dark 변수로.
- **시각 산출물(HTML/SVG/canvas/차트/UI)은 헤드리스 DOM 검증에서 멈추지 말고 raster 스크린샷으로 루프를 닫아라.** lightpanda는 JS·DOM은 주지만 *렌더링을 안 한다* — 레이아웃·라벨 겹침·색·톤·곡선 모양은 안 보이고 `getComputedStyle`도 background 단축·`--var` 커스텀속성에 빈 값을 준다. 따라서 헤드리스는 로직/구조만 확인하고, 시각 정합성은 실제 raster가 필요. 절차: 편집 → `Google Chrome --headless=new --screenshot=out.png --window-size=W,H "file://<urlencoded-abs-path>"` → PIL로 관심영역 crop+1.5× 확대 → Read 툴로 이미지 확인 → 타깃과 대조 → 반복(전체 페이지 샷은 디테일 판단 불가, 반드시 crop). 한글 파일명은 `urllib.parse.quote`. 사용자에게 매번 캡처를 시키지 말고 내가 직접 본다 — "검증했다"는 곧 "렌더해서 눈으로 봤다".
- **적대 검증 fan-out(deep-research류) 결과의 vault화는 "옵션별 카드(micro-row) → 확정 findings 표(confidence·vote) → 무너진 가정 red killbox → open questions" 4단이 강력하다.** 여러 각도로 흩어 수집·검증한 결과를 한 문서로 모을 때: ① 비교 대상(방향·옵션·후보)마다 좌보더 색 카드 + 고정 라벨 micro-row(예: incumbent·빈틈·간판·아키타입)로 N개를 시각 정렬, 각 행 끝에 `vote`(3-0 등) pill로 검증 강도 노출. ② 확정 findings는 `finding·conf·vote` 3열 표로 압축하고 핵심 행만 bold. ③ **가장 중요한 장치 — "무너진 가정"을 1급 섹션(red `killbox`)으로 박는다**: 적대 검증의 진짜 산출물은 *확정된 사실*이 아니라 *분석자 자신의 prior가 깨진 목록*이다(0-3/1-2로 기각된 것). 이걸 본문 하단 각주가 아니라 독립 빨강 박스로 올려야 미래의 자기가 죽은 가정을 되살리지 않는다. ④ open questions로 닫아 "검증 못 한 것"을 명시(특히 1차 측정 데이터 미확보 항목). 같은 주제 전략 문서가 따로 있으면 footer cross-link로 "전략 SSOT ↔ 검증 산출물"을 쌍지운다.
- **judge-panel 합성 설계 산출물(목차·기획안 등 "N개 후보 → 채점 → 우승안 합성")의 vault화는 "채점 ranking 표 → positioning 카드 → 항목 카드(hook + 하위 리스트) → 설계 노트(이식 출처 + 남은 리스크)" 4단이 정형이다.** 적대검증 fan-out(앞 룰)이 *사실 검증*이면 이건 *설계 합성*이라 골격이 다르다 — ① 후보별 overall 점수를 우승안에 `win` 행 배경 + `tag`(베이스/페이로드)로 표시한 ranking 표를 먼저 박아 "왜 이 합성인가"를 첫 화면에 답하고, ② 차별 포지셔닝·핵심 스탠스는 본문 앞 `.pos` 카드(좌액센트 보더, `.k` 라벨) 한두 문단으로 못박는다. ③ 본문 항목(장·절·단계)은 각 카드 = `제목 + 한 줄 hook(italic muted) + 하위 리스트` 고정 리듬으로 N개를 align, 완결 arc가 있으면 카드 흐름 위에 `arc strip`(단계→단계) 한 줄. ④ 가장 재사용 가치 큰 닫기 = **"설계 노트"** — 우승 각도·*무엇을 어느 후보에서 이식했나*(graft 출처)·경쟁자 대비 wedge·남은 리스크를 명시해, 미래의 자기가 "왜 이 구조지"를 재론하지 않게(합성은 출처를 잃으면 blackbox가 된다). 전략 SSOT 문서가 따로 있으면 footer cross-link, 짝 산출물(원고 manuscript 등)도 경로로 명시.
- **vault 산출물을 공개 정적 사이트로 발행할 땐 "연구 아티팩트 → 제품면" 정리 + 호스트 함정 체크가 정형이다.** 자기완결 HTML을 공개 도메인에 올릴 때: ① **별도 레포로 분리** — 개인 monorepo 안에 두면 호스트의 git-app이 그 레포 *전체*(민감 이웃 포함) 읽기 권한을 갖는다(폴더 단위로 못 좁힘). ② 발행 전 **내부 chrome 제거**: 다른 노트로의 상대링크(공개 시 404), retro/버전 회고 배너(내부 전략 노출), PII. ③ **OG/twitter 카드 메타 필수** — 밈·비주얼 제품은 공유 썸네일 없으면 죽은 링크. 카드는 아티팩트 히어로를 스크린샷해 생성. ④ 면책 유지. **정적 호스트(Vercel류) 함정:** 신규 프로젝트는 Deployment Protection 기본 ON(401)→공개하려면 끄기; 커스텀 apex는 호스트가 보고하는 '현재 감지된 A레코드'가 목표가 아니라 호스트의 *recommended* IPv4가 목표; DNS는 권위 NS에 `dig @ns`로 캐시 우회 검증하고 인증서 SAN이 apex를 실제 커버하는지 확인(www만 먼저 발급되면 수동 발급 강제).
