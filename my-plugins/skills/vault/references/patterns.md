# Vault — Accumulated Layout Patterns

Distilled, reusable HTML layout patterns from past vault saves, organized loosely by content type.
SKILL.md "HTML creation" greps this file to reuse a matching layout; "Reflection" appends new patterns here (1–2 lines each).

Grep by content-type keyword, e.g.:
```
grep -iE 'self-check|카탈로그|how-to|입문|snapshot|redesign|grilling|reference' references/patterns.md
```

---

- **설계 노트는 Q&A 카드 형태가 효과적.** 결정 사항이 많을 때 `<div class="decision">` 블록으로 질문/답 구조를 만들면 나중에 읽기 쉽다.
- **index.html은 `<title>` 태그 파싱으로 재생성.** 파일마다 별도 메타데이터 없이 title만으로 충분히 깔끔한 아카이브 페이지를 만들 수 있다.
- **설정/스냅샷류는 grid + key-value 카드가 적합.** 정량 데이터(boolean 토글, 값, 모드)는 텍스트로 풀어쓰지 말고 `auto-fill minmax(220px, 1fr)` 그리드의 KV 카드로 배치하면 한 눈에 스캔 가능.
- **이벤트/파이프라인 흐름은 `label → arrow → target` 구조.** Hook chain 같은 순차적 흐름은 좌측 라벨(이벤트명) · 중앙 화살표 · 우측 타겟의 단순 3열로 시각화하면 다이어그램 없이도 흐름이 읽힌다.
- **섹터/도메인 입문 글은 `TL;DR → 30초 사전지식 → 기술 분류표 → 촉매(데이터 포함) → 종목 카드 → Bull/Bear 2열 → 운용 포인트` 순서가 잘 먹는다.** 사용자가 비전문가일 때, 그가 이미 아는 도메인(예: SWE → ISA 비유)으로 분류 체계를 매핑하면 이해 속도가 크게 올라간다.
- **연속 섹터 정리는 마지막에 "이전 글과의 비교 표"를 붙여라.** vault에 이미 비슷한 카테고리 글이 있으면, 새 글 마지막에 `축 × 두 섹터` 형태의 6~8행 표로 연결하면 독자가 새 정보를 기존 멘탈 모델에 끼워 넣기 쉽다.
- **기존 vault 항목 업데이트는 새 파일 대신 inline addendum.** 같은 주제의 후속 분석은 새 날짜 파일을 만들기보다 원본에 `<hr>` + 새 `<header>` 블록으로 이어 붙이면 컨텍스트가 한 곳에 모인다. 날짜는 달라도 **같은 날(same-day) 후속도 addendum이 맞다** — 거의-중복 새 파일을 만들지 말 것. 효과를 극대화하려면 (1) 섹션 numbering을 원본에서 이어받고(예: 01~05 → 06~), (2) 원본의 semantic color class(`.verdict/.vrow`, `tr.variance/.opportunity/.ruin`, `.calc .pos/.neg`)를 그대로 재사용하면 addendum이 bolt-on이 아니라 한 문서로 읽힌다. index.html 재생성 불필요(제목이 여전히 맞으면 손대지 말 것), 커밋은 그 한 파일만 스테이징 + 메시지에 "addendum" 명시.
- **index.html은 통째로 Write하지 말고 항상 Read 후 Edit으로 surgical 갱신.** 사용자가 index.html을 직접 커스터마이즈하는 경우가 있다(badges/tags/month grouping/카운트 등). 매번 통째 재생성하면 사용자 작업이 사라진다. 새 항목 추가는 (1) Read로 현재 구조 확인 (2) 적절한 위치에 `<li>` Edit으로 삽입 (3) 카운트 필드 갱신. "regenerate"는 fallback일 뿐이고 default는 incremental edit.
- **참고/리퍼런스 doc는 featured 카드 + 보조 카드 + "★ why" 주석 패턴이 효과적.** 외부 레포·도구를 다수 나열할 때, 핵심 1~2개는 `.featured` 클래스로 강조(그라데이션 배경, 컬러 보더)하고 나머지는 plain card. 각 카드 하단에 `★ ` 프리픽스로 "왜 주목할 가치 있는가" 1줄 주석을 별색(녹색 톤)으로 붙이면 사용자가 자기 컨텍스트와 매칭해서 빠르게 스캔 가능.
- **"넓은 카탈로그 doc" + "특정 소스 필터 doc"의 쌍은 자연스러운 시리즈를 만든다.** 한 주제로 broad reference를 한 번 만들고, 이어서 같은 주제를 특정 소스(news.hada.io / HN / arxiv 등)로 필터링한 후속 doc을 만들면, 두 doc이 footer에서 서로 cross-link되어 "전체 풍경 → 국지적 시그널" 두 단계 탐색이 가능. 후속 doc은 source ID(예: GeekNews topic id)를 leftmost 컬럼으로 노출하면 tabular feel + 빠른 navigation.
- **진행 중 grilling/planning 노트는 "decided" vs "open" 두 상태 카드로 시각 구분.** 좌측 보더 색(decided=녹색, open=주황)과 `.dec-tag` 의 status 라벨로 닫힌 결정과 미해결 질문을 한 페이지에서 분리. 각 카드는 `Q번호 · subcategory · status` 메타 + 질문 + `→ <span class="pick">선택지</span>` + 근거 구조. 세션이 끝나기 전에도 중간 상태를 vault에 남길 수 있어 멀티세션 워크플로우와 잘 맞음. 추가로 결정 카드 위에 KV 그리드 `프로필 스냅샷`을 두면 후속 독자(또는 미래의 자기)가 컨텍스트를 빠르게 복원.
- **Redesign/iteration류 노트는 "before↔after ASCII 비교 + iteration 카드(변경+self-critique) + 핵심 코드 스니펫" 패턴이 효과적.** 결과만 나열하지 말고 라운드별 결정 사유와 그 라운드의 부족함(self-critique)을 카드로 분리하면 progression이 추적 가능 — 다음 iteration의 trigger를 명시적으로 남긴다. 디자인 노트 자체를 노트가 설명하는 그 디자인 시스템으로 만들면(예: light paper + Pretendard 본문 + Mono 메타) 메타-설명력이 올라가고 reference 구현으로도 작동.
- **운영 절차의 비효율은 single-source-of-truth + JS derive로 흡수한다.** index.html 같은 "데이터 + view" 페이지는 chip 카운트·outline·month section·NEW pill 등 derive 가능한 모든 필드를 정적으로 박지 말고 JS가 한 source(`<ul id="src">`)에서 계산해 채우게 한다. 운영 절차가 "8지점 갱신 → 1줄 추가"로 축소되고, 카운트 불일치 risk가 영구적으로 사라진다. 사적/소규모 페이지면 JS-off fallback은 ul 한 개라 거의 무료. 통째 Write 금지(learning #79)는 여전히 유효하지만, surgical edit의 대상이 더 작아진다.
- **메타 분석/도입 결정 노트는 "rules 그리드 + 대조 테이블 + decided 카드 + 액션 체크리스트" 조합이 강력.** 외부 도구·라이브러리·스킬을 우리 setup에 흡수할지 검토할 때 ① 외부 규칙을 카드 그리드로 압축(`auto-fit minmax`), ② 우리 현재 상태와 규칙 단위 ✓/⚠ 대조 테이블, ③ `decided` 카드(좌측 녹색 보더 + `.dec-tag`)로 결정 + 근거 한 문단, ④ 즉시 실행 가능한 명령 포함 액션 체크리스트(번호 매김) — 네 영역이 한 페이지에서 "외부 패턴 → 진단 → 결정 → 실행"의 완결 사이클을 보여준다. write-a-skill 같은 메타스킬 검토에 특히 잘 맞고, 후속 도입 검토 노트의 템플릿으로 재사용 가능.
- **여러 primer를 묶는 종합 doc은 "티어 색상 시스템 + 가로 allocation bar + thesis-breaking 체크리스트"로 구조화.** 카테고리(Core/Satellite/Speculative)에 일관된 색(녹/노/빨)을 CSS 변수로 박고 ① 종목 카드 좌측 보더 ② pill 라벨 ③ 가로 stacked bar 모두 같은 색으로 동기화하면 분류가 시각적으로 즉시 읽힘. 끝부분에 "이 조건이 트리거되면 thesis 깨짐" empty-checkbox 리스트(`☐` ::before)를 두면 단순 분석을 넘어 운용 가능한 문서가 됨. Footer에 베이스가 된 vault 항목들을 cross-link하면 다음 독자가 출처를 한 번에 추적 가능.
- **카탈로그 doc은 분류 축이 가치를 좌우한다 — "사용자 셋업과의 매칭" 축이 가장 강력.** awesome-list 형태로 "공식 vs 큐레이션 vs 회사 mega-pack"으로 자르면 단순 카탈로그에 그치지만, "structure-match · dual-env match · 도메인 일치"처럼 사용자의 본인 셋업과의 거리로 자르면 actionable해진다. doc 끝에는 반드시 <strong>"옮겨올 implementable picks" 섹션</strong>을 둬서 흥미→행동 다리를 놓을 것 (각 픽에 `<code>아이템</code> <span class="from">from 원본</span>` 구조로 출처와 함께). 사용자가 "이런 거 원했어"라고 다른 축을 신호하면 즉시 rewrite — 카탈로그는 잘못 분류되면 거의 무가치.
- **개념/용어 해부 explainer는 "N개 변종에 일관된 색을 박고 여러 표현(표 → 질문 카드 → verdict 행)에 동기화"하는 패턴이 강력.** 한 단어(예: "risk")가 사실 N개의 다른 정의로 쓰임을 보일 때, 각 변종(variance/opportunity/ruin)에 CSS 변수 색을 하나씩 배정하고 ① 분류 표의 행 ② "각자 묻는 질문" 카드 ③ 최종 verdict 행에 같은 색을 반복하면 독자가 색만으로 변종을 추적. 정량 비교(앙상블 vs 시간 평균 같은)는 `.calc` 모노스페이스 블록에 색칠한 숫자로 단계별 노출하면 수식 없이도 직관이 전달됨. 마지막 callout `한 줄 요약`으로 closure. 비전문가용 개념 글의 재사용 템플릿.
- **"verbatim 캡처 + 해부" 혼합 doc은 fidelity 경계를 시각적으로 표시하라.** 시스템 프롬프트·설정처럼 일부는 원문 그대로, 일부는 설명인 문서는 ① verbatim 블록에 `vtag`(static=녹 / dynamic=주황) 라벨 + 좌측 보더 색으로 "이건 실제 텍스트"임을 명시하고, ② 동적/주입 부분(사용자 본인 파일 — CLAUDE.md, MEMORY.md, gitStatus 등)은 통째 재수록 말고 출처 표로 가리킨다(bloat 방지 + 소유권 존중). 조직 축은 7개 나열보다 직교하는 "static vs dynamic" 한 줄이 강력 — 색-코딩이 표·태그·보더에 일관 동기화되면 독자가 색만으로 구운 것 vs 주입된 것을 추적. 개념 비교(harness vs persona 등)는 2열 `.card`, 위험 원칙(blast radius)은 `.callout` 1개로 닫음.
- **테제 검증/de-bias(self-check) 노트는 "테제 callout → SUPPORTS/CONTRADICTS 색코딩 카드 → perception-bias 메타 callout → 보정 테제 + 할인율 split-bar → 증거품질 그리드" 구조가 강력.** 살아남은 발견(녹 보더 `.s`)과 무너진 발견(빨 보더 `.c`)을 `SUPPORTS`/`CONTRADICTS` pill로 라벨링하고 각 카드에 1차/2차 출처를 `.src` 모노 라벨로 노출하면 신뢰도가 함께 읽힌다. 핵심 장치 둘: ① **할인율 split-bar** — 원 테제를 몇 % 깎는지를 `유지%`(녹)/`폐기%`(빨) 두 색 바로 수치화(단순 결론보다 강한 closure). ② **증거품질 그리드** — `firm`(1차)/`soft`(2차 미검증)/`disputed`(인과 논쟁)를 상단 보더 색이 다른 KV 카드로 분리해 "근거 자체도 검증했음"을 보여줌. fan-out 수렴(아래)이 발견 *수집*용이면, 이건 발견을 *판결*하는 용도.
- **self-check 노트는 검증 후 떠오른 "후속 질문"을 같은 색 vocab으로 한 섹션 더 흡수할 수 있다.** 보정 테제 다음에 사용자가 던진 연관 직관(예: "고립된 헤비유저는 → 오타쿠?")이 있으면, 새 파일 대신 마지막에 `N · 후속` 섹션을 붙여 ① 그 직관이 fear-confirming 서사일 가능성을 `.callout.meta`로 먼저 해체하고 ② 살아남는 변수(녹 `.s`)와 진짜 위험(빨 `.c`)을 같은 SUPPORTS/CONTRADICTS 카드로 판결한다. 검증된 본문과 후속 탐색이 한 문서에 모이고, "입증 vs 그럴듯함" 구분이 카드 색으로 유지된다.
- **Fan-out 결과 수렴 doc은 "클러스터 strip → 합집합 tier 카드 → 키워드 빈도 표 → 약점 체크리스트" 4단 구조.** 여러 sub-agent가 횡으로 흩어 수집한 raw output을 vault에 다시 모을 때: ① flex-proportional `.strip`으로 클러스터별 발견 수 분포를 즉시 보여주고 ② **클러스터 경계를 무시하고 S/A/B tier로 재정렬**해 큰 카드로 노출(클러스터는 우측 colored tag로만 남김) ③ 4 클러스터 키워드를 합산한 frequency 표에 "본인 매칭(강/중/약)" 컬럼을 붙이면 객관적 신호와 자기 평가가 동시에 보임 ④ 약점 보완 체크리스트로 closure. tier 카드 안에 `<span class="tag">키워드</span>` pill 1-2개를 메모 앞에 박으면 fit 근거가 즉시 노출. 다음 단계(Phase N)로 넘어가는 시사점을 footer 직전에 짧게 정리. 사용자 의사결정 노트(`-plan.html`)와 결과 수렴 노트(`-shortlist.html`)는 한 쌍으로 짝지어 두면 grilling 세션의 입력/출력이 vault에 보존됨.
- **절차형 how-to/생활 가이드는 "상단 N단계 flow strip → 단계별 섹션 → safe/caution/danger 3색 동기화 → 끝에 액션 체크리스트" 패턴이 강력.** 빨래·요리처럼 비전문가 대상 실전 가이드는 ① 상단에 numbered flow strip(`step → step`)으로 전체 흐름을 먼저 보여주고 위험 단계는 색으로 강조, ② 핵심 결론은 맨 위 `key` callout 한 줄로 박고(예: "새 옷 따로·세제 적게·바로 꺼내라"), ③ "이건 안전 / 주의 / 망가짐"을 녹·주황·빨 3색으로 카드·note·인라인 텍스트에 일관 동기화하면 독자가 색만으로 do/don't를 추적, ④ 맨 끝 `☐` 체크리스트로 "오늘 바로 따라 하기"를 닫는다. 금융/CC 외 도메인(집안일 등) 첫 진입 시에도 잘 작동하는 범용 템플릿.
- **다중 카테고리 "핸드북"은 병렬 fan-out 리서치 → 단일 self-contained doc 합성이 효율적.** 5개+ 생활/도메인 카테고리를 각각 sonnet sub-agent에 동시 위임(WebSearch 포함)해 구조화 브리프를 받고 한 HTML로 합성한다. ① 상단 sticky `nav.toc`(앵커 칩)로 섹션 점프 ② "우선순위(impact×빈도)" overview strip을 본문 앞에 둬 "어디부터" 답을 먼저 줌 ③ 모든 섹션이 같은 절차형 패턴(위 학습) 반복 — 단일 semantic 3색(ok/warn/bad)·카드·표·`☐` 체크리스트를 동일하게 써서 N개 이질 도메인이 한 시스템으로 읽힘 ④ 섹션마다 끝에 카테고리 마스터 체크리스트로 닫음 ⑤ 같은 시리즈의 단일 주제 doc(예: 빨래 가이드)이 있으면 cross-link. 브리프는 verbatim 재수록 말고 high-value만 distill(표 compact). 페르소나·지역 특화(특정지역 분리수거, 2026 세무 등)는 sub-agent 프롬프트에 명시해 현재성 확보.
- **grilling 노트가 낯선 도메인 교육까지 겸할 땐 "단일 fit-verdict 팔레트(go/sub/no)"를 이질적 섹션 전체에 동기화하라.** 사용자가 "이름만 알지 비교 대상 모름"인 분야의 구매/선택 결정에선, 한 문서가 ① 결정 트레일(Q→pick→why 카드) + ② 도메인 지형도 카탈로그 표를 동시에 담으면 "내 결정 기록 + 분야 입문서" 이중 역할을 한다. 핵심 장치: 같은 3색(녹=정중앙/추천, 주황=보조, 빨=패스)을 (a) tier 리스트 좌측 보더 (b) 카탈로그 표의 마지막 "나한테" 컬럼(`.fit.go/.sub/.no`) (c) how-to 루트 카드에 반복하면 독자가 색만으로 "맞음/보류/스킵"을 전 섹션에서 추적. 카탈로그 표는 객관 특징 컬럼 + **"X ↔ Y 비교 축" 한 줄 mnemonic(`.axes`)**으로 닫으면 "누가 누구랑 경쟁하나"가 압축됨. 결정의 출발점(예: 원래 조사하던 브랜드)은 끝에 `참고` 섹션 + `.callout.amber`로 보존해 전체 arc를 남긴다.
- **결론이 외부 제약으로 "뒤집히는" 의사결정 노트는 보정 테제와 제약 수렴을 둘 다 green으로 나란히 박아 backsliding이 아님을 시각화하라.** self-check가 테제를 깎았는데(예: "메신저앱이 설계상 우월") 그 다음 하드 제약(회사망 차단)이 원래 선택으로 다시 데려가는 경우, 보정 테제(green thesis 블록)와 "제약이 고른 결과"(green callout)를 연속 배치하면 독자가 "동기적 합리화로 되돌아간 것"이 아니라 "환경이 결정했다"를 읽는다. 그리고 노트가 **빌드로 끝나면 묘사하지 말고 실제 검증 출력(stdout/JSON 왕복)을 `<pre>`에 verbatim + 행 끝 `✓` 주석**으로 박아라 — "콜백 계약대로 동작"을 말로 하는 것보다 실제 요청/응답 페어가 압도적으로 설득력 있다. de-bias(#123) + done/open(#128) + 검증 출력의 3종 결합은 "탐색→판결→빌드" 한 사이클을 한 문서에 닫는다.
- **self-check 노트가 실제 변경(action)으로 이어질 땐 "판결 → scope 뱃지 스냅샷 → done/open 체크리스트"로 루프를 닫아라.** de-bias 검증(thesis/할인율 split-bar/증거품질, learning #123)만으로 끝내지 말고, 그 결론으로 무엇을 바꿨는지를 KV 스냅샷 그리드로 잇되 각 항목에 **전파 범위 뱃지**를 단다 — `repo`(녹, version-controlled → 다른 머신에 따라옴) vs `machine-local`(주황, 머신별 → 수동 복제 필요). dual-env(개인+회사) 셋업에선 "이 변경이 다른 머신에 따라오나?"가 가장 자주 묻는 질문이고, 뱃지 하나로 즉시 답된다. 끝의 체크리스트는 `☑ done`(녹) / `☐ open`(주황)으로 갈라 "검증·적용 완료 vs 관찰 필요"를 분리하면 멀티세션에서 다음에 뭘 확인할지가 보존된다. 이벤트 흐름(hook→BEL→플래그→색)이 있으면 `label → arrow → target` 행(#108)으로 같이 시각화.
- **코퍼스/시스템 감사 doc은 "hero stat 카드 → 벤치마크 열 스코어카드 → 자기-반증 인용 → ROI순 promote된 triage" 구조.** 사용자 본인의 데이터(노트 그래프·설정·로그)를 스크립트로 전수 채점한 결과를 남길 때: ① 헤드라인 4개 수치를 `.stat`(big mono 숫자 + bad/ok/warn 색) hero 그리드로 punchy하게 열고, ② 스코어카드 표는 측정값 옆에 **"건강 기준" 열 + 색-코딩 판정**(`.j.bad/.ok/.warn`)을 둬서 "좋은가 나쁜가"가 즉시 읽히게 한다(맨숫자만 나열 금지). ③ 가장 강력한 장치 — **사용자 자신의 산출물을 인용**해 "선언한 방법 vs 측정된 실천"의 괴리를 `.callout`에 박으면 진단이 반박 불가가 된다(머릿속이 아니라 파일이 증거). ④ triage는 문제를 나열하지 말고 **최고 ROI 버킷을 `.tier.hot`(채운 녹색 bg)으로 맨 위에 promote**(나머지는 warn/bad). 단일 stacked `.bar`로 코퍼스 구성(고립% vs 연결%)을 한 줄 시각화하면 hero를 보강. 프로세스 측: "전수검수"류는 N개 통독 금지 — 스크립트로 전 코퍼스를 구조 채점(싸고 100% 커버) 후 플래그된 것 + 방법론-정의 노트만 정독.
- **여러 이질적 이론을 한 문서에 묶을 땐 "하나의 이항 축"에 2색을 박고 모든 표현에 관통시켜라 — 그래야 이론 나열이 아니라 한 논증으로 읽힌다.** #29가 한 단어의 N개 변종에 색을 배정하는 거라면, 이건 *서로 다른 분야*(화용론 locution/illocution · 메타커뮤니케이션 content/relationship · RSA 정보효용/사회효용)가 사실 같은 이항 축(표면 vs 진짜)의 다른 이름임을 색으로 드러내는 것. 상단에 2채널 `.legend` 카드(`.s` 파랑=표면 / `.r` 주황=진짜)를 박고, 그 두 색을 ① 유형 카드의 pivot 보더 ② worked example의 `.surf/.rel` 인라인 ③ 이론 비교표 셀 ④ RSA 효용 분해 stacked bar까지 반복하면, 독자가 색만으로 "지금 이 개념이 어느 채널 얘긴지"를 분야가 바뀌어도 추적한다. 그리고 **재귀/중첩 모델(RSA의 L1⊃S1⊃L0, theory-of-mind, 콜 스택)은 산문 말고 중첩 DOM 박스로 그려라** — `.frame` div와 `border-left` 색 깊이로 실제로 박스를 포개면 "시간 순서가 아니라 깊이"라는 핵심 정정이 시각적으로 자명해진다(중첩 표 한 줄이 문단 셋을 이긴다). 곤란한 질문→커뮤니케이션론 같은 "직관이 알고 보니 학문" 류 explainer의 표준.
- **사용자가 실제로 "적용할" 다중 추천 카탈로그는 항목마다 `붙여넣을 수 있는 config 스니펫 + 명시적 기대효과` 쌍을 강제하고, 상단 요약표에 scope 열을 둬라.** "도구 N개 추천"은 분류 축(#28)·단일 도입 결정(#26)과 다르다 — 핵심은 *항목별 실행가능성*. 각 카드를 고정 라벨 리듬(`왜 너에게` → `예시`(실제 .gitconfig/.zshrc/settings.json 스니펫을 `<pre>`에 verbatim) → `기대효과`(결과 문장, 녹) → `주의`(충돌·trade-off, 주황) → `source`)으로 통일하면 긴 카탈로그가 스캔된다. 추천은 *정확한 스니펫*과 *payoff 문장*이 둘 다 있어야 actionable — 둘 중 하나만 있으면 "흥미롭지만 안 함"으로 끝난다. 맨 위 요약표는 `# · 도구 · 한 줄 기대효과 · scope 뱃지`(stow=녹/machine-local=주황, #38 재사용) 4열로 — 독자가 detail 카드 들어가기 전에 "뭐가 있고 어디에 적용되나"를 먼저 본다. 도메인별(터미널/tmux/CC) group 헤더로 카드를 묶고, 채택 안 한 후보는 `honorable mention`에 이유와 함께 보존(왜 *안* 골랐나도 정보). 같은 두 축(예: 익명성/검열)을 이질적 도메인(검색엔진·VPN) 각각에 적용하는 종합 문서에선, ① 각 축에 CSS 변수 색을 하나씩 박아 inline `.badge`로 만들고 모든 질문 헤더(`Q1 … <badge>익명성</badge>`)에 반복하면 독자가 "이 발견이 어느 축 소속인지"를 색만으로 추적 — 도메인이 바뀌어도 축 색은 불변이라 cross-domain 패턴(예: "독립/자가호스팅은 검열엔 강하나 익명성엔 약함")이 색의 반복으로 드러난다. ② 그 위에 **축과 직교하는 신뢰도 pill**(`확인됨`=녹 1차/논문·법원, `주장`=주황 2차매체, `추정`=회색)을 개별 claim 끝에 달면, sub-agent가 모아온 출처 품질이 footnote 추적 없이 인라인으로 읽힌다 — 리서치 종합문서가 "결론"과 "그 결론의 증거강도"를 한 줄에 동시 노출. 끝은 `축 색 그라디언트 axis-rule`로 두 축이 섞이는 교차 시사점 섹션을 닫고, footer legend에 두 색뱃지+신뢰도 정의를 재수록. 두 sub-agent 병렬 위임 → 한 문서 종합 시 표준 템플릿.
- **사용자의 *다른 시스템*(PKM·dotfiles·타 repo)에 대한 의사결정 노트는, 그 시스템에 실제 짝 artifact를 만들어 서로 cross-link해 "결정을 dogfood"하면 closure가 압도적으로 강해진다.** "X를 어떻게 조직할까" 결정을 vault에 글로만 남기면 또 하나의 안 읽는 산출물이다. 대신 결정이 규정하는 그 행위를 즉시 1회 실행 — 예: "vault=literature, Zettel=permanent, 다리=vault항목→org노트1개" 결정이면, 진짜 org permanent 노트 하나를 사용자 컨벤션(org-roam `:ID:`+`#+title:`, 그의 best 패턴인 Related Hubs/Notes)대로 만들고 실제 이웃 ID로 링크 + vault를 `[[file:]]`로 인용, vault 쪽도 그 org 노트명을 명시해 양방향 참조. 노트가 자기 thesis의 proof-of-concept가 된다. 시각 장치: ① 역할 분담은 2-col `.role`(시스템별 top-border 색) + 정확한 출처 프레임(예: Ahrens literature→permanent) 매핑, ② 다리는 `label → arrow → target` flow 행(#16) + "대부분 → 0개" dim 행으로 *안 하는 게 정상*임을 명시, ③ 의존 방향은 `A → B(ok 녹) / B → A(no 빨 line-through)` 한 줄 `.dir`, ④ A/B 갈림길은 `.opt.pick`로 고른 쪽만 채우고 근거(사용자의 과거 성공 경험 인용)를 callout으로. 단, 짝 artifact는 **seed로 명시**(주석으로 "네 말로 다시 써라")하고 사용자 private repo는 **자동 커밋 금지** — 생성만, 커밋은 본인.
